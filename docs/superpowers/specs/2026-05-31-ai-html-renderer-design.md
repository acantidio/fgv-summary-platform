# AI HTML Renderer — Design Spec
**Date:** 2026-05-31
**Status:** Approved

---

## Overview

Add a third layer to the content pipeline: an AI-powered HTML renderer that takes the enriched markdown for each subject and generates a beautiful, subject-specific HTML page. This replaces `build.js` entirely. The new script (`render.js`) calls Claude Opus to produce rich, adaptive HTML where the visual structure matches the cognitive structure of the content.

---

## Pipeline

```
content/[slug].md
    ↓  enrich.js (Opus) — unchanged
content/enriched/[slug].md
    ↓  render.js (Opus) — NEW
docs/[slug]/index.html
```

The hub page (`docs/index.html`) is still generated by a lightweight templated function inside `render.js` — no AI involved.

---

## Architecture

### `render.js` exports

| Export | Description |
|---|---|
| `renderSubject(slug)` | Reads `content/enriched/[slug].md`, calls Opus, writes `docs/[slug]/index.html`, then calls `buildHub()` |
| `buildHub()` | Reads frontmatter from all `content/*.md` files, generates `docs/index.html` (no AI, same card grid as today) |
| `renderSite()` | Calls `renderSubject()` for every slug that has an enriched file, then `buildHub()` |

All three functions are exported so `test.js` can import and test them directly.

### `renderSubject(slug)` flow

1. Read `content/enriched/[slug].md` — throw a descriptive error if the file is missing (prompt user to run `npm run enrich -- [slug]` first)
2. Parse frontmatter with `gray-matter` to extract `title`, `color`, `status`, `description`
3. Build the user message: frontmatter fields as structured context + the enriched markdown body
4. Call Claude Opus (`claude-opus-4-8`, `max_tokens: 16000`) with the system prompt
5. Write the raw response (a complete `<!DOCTYPE html>` document) to `docs/[slug]/index.html`, creating the directory if needed
6. Call `buildHub()` to keep the hub in sync

### `buildHub()`

Reads all `content/*.md` files for their frontmatter, generates `docs/index.html` using the same templated card grid as the current `build.js` implementation. This function is synchronous and has no AI dependency — it can run in CI if ever needed.

### CLI

```bash
npm run render -- [slug]       # render one subject + regenerate hub
npm run render -- --all        # render all subjects with enriched files + hub
```

`--all` renders subjects sequentially to avoid rate limits and produce readable log output.

## The Render Prompt

### Part 1 — Role & mission

Sets the model as a world-class educational interface designer. The framing is "transform this learning content into a beautiful, subject-specific HTML page that makes the knowledge easier to absorb" — not "convert markdown to HTML."

### Part 2 — Design language (hardcoded in prompt)

The full CSS variable palette is embedded verbatim so the model always uses exact tokens — never invents new hex values.

**Fonts:**
- `DM Serif Display` — display headings
- `DM Sans` — body text
- `JetBrains Mono` — labels, badges, mono

**Color tokens:**
```
--bg: #FAF9F6          --bg-card: #FFFFFF       --bg-surface: #F3F1EC
--text-primary: #1A1A18  --text-secondary: #5C5B56  --text-tertiary: #8A8985
--border: #DDD9D0       --border-light: #ECEAE4

Purple: #EEEDFE / #7F77DD / #534AB7 / #3C3489
Amber:  #FAEEDA / #BA7517 / #854F0B / #633806
Teal:   #E6F4F1 / #1D9E75 / #0F6E56 / #085041
Blue:   #E6F1FB / #378ADD / #185FA5 / #0C447C
Rose:   #FDE8EC / #C81E3A
```

The subject's `color` field from frontmatter drives the page accent: top border, badge, and highlight color.

### Part 3 — Adaptive component rules

The model must match visual form to cognitive structure. Explicit mappings:

| Content pattern | Visual component |
|---|---|
| 2×2 analysis (SWOT, quadrants) | Color-coded 2×2 grid |
| Named framework with 4–6 pillars (BSC, Diamond) | Card grid or labeled diagram |
| Sequence of steps / phases | Numbered stepper with connectors |
| Forces / actors around a center (Porter) | Hub-and-spoke layout |
| Objectives + metrics (BSC Painel) | Structured table with column headers |
| Single concept / definition | Definition card: term + explanation |
| Summary (`[!SUMMARY]`) | Hero callout at page top — larger, more prominent treatment |
| Exam alert (`[!EXAM]`) | Amber alert banner, visually prominent |
| Key concept (`[!KEY]`) | Purple definition card |
| Self-test question (`[!RECALL]`) | Distinct teal question card, visually separated |

**Rule:** a bullet list is a last resort, not a default. If content has structure, render that structure visually.

### Part 4 — Output rules

- Complete `<!DOCTYPE html>` document with all CSS in a single `<style>` block
- Google Fonts import is the only permitted external dependency
- Zero JavaScript — fully static output
- Back link `← FGV MBA` pointing to `../`
- Sticky nav with anchor links to each major section
- Responsive — mobile breakpoint at 640px
- All text in Portuguese
- Every fact from the input must appear in the output — no truncation, no invention

### Part 5 — Anti-patterns (explicit prohibitions)

- Do not use bullet lists for content that has visual structure
- Do not use generic left-border callouts for everything — the four callout types each get a distinct visual treatment
- Do not invent colors outside the token system
- Do not add placeholder, example, or invented content
- Do not reference tooling, file names, or internal pipeline details in the output

## CI / CD

Unchanged in structure. GitHub Actions runs `npm test` + deploys committed `docs/`. The render step is local-only, same as `enrich.js`. Neither enrich nor render runs in CI.

---

## Implementation Notes

- Model: `claude-opus-4-8` (same as enrich.js — always latest Opus, check before implementing)
- `max_tokens: 16000` (subject pages can be substantially longer than enriched markdown)
- `.env` handling: same pattern as `enrich.js` — read from `.env` at project root, never committed
- Re-render a subject: `npm run render -- [slug]` (idempotent, overwrites existing file)
- Error on missing enriched file: `Error: No enriched file found for "[slug]". Run: npm run enrich -- [slug]`
